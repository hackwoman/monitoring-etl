# CMDB 优化方向 — 大厂 CMDB 产品调研与对比分析

_日期：2026-04-01_
_来源：蓝鲸 CMDB (腾讯)、WeCMDB (微众银行)、嘉为蓝鲸、博睿 ONE_

## 1. 大厂 CMDB 产品核心特点

### 1.1 蓝鲸 CMDB (Tencent BlueKing)

**架构：微服务四层**
```
API Gateway
  ↓
业务场景层（贴近业务的场景服务，无状态，去中心化）
  ↓
资源管理层（资源抽象 controller，代理所有操作）
  ↓
存储层
```

**核心能力：**

| 能力 | 说明 |
|------|------|
| **自定义模型** | 内置模型可扩展属性，可随时新增模型和关联关系 |
| **拓扑化主机管理** | 主机基础属性 + 快照数据 + 归属关系 |
| **组织架构管理** | 可扩展的基于业务的组织架构 |
| **模型管理** | 业务、集群、主机等内置模型 + 自定义模型 |
| **进程管理** | 基于模块的主机进程管理 |
| **事件注册与推送** | 基于回调方式的事件注册与推送 |
| **通用权限管理** | 基于用户组的灵活权限 |
| **操作审计** | 用户操作行为审计与回溯 |
| **数据自动发现** | 自动发现资源变更 |
| **变更事件主动推送** | 数据变更通知下游 |

**设计亮点：**
- 资源原子管理 vs 业务场景分离 — 场景层复用资源管理层
- 新增场景服务不影响现有服务
- 蓝绿发布 / 金丝雀发布支持

### 1.2 WeCMDB (微众银行)

**核心特点：**

| 能力 | 说明 |
|------|------|
| **配置模型动态扩展** | 灵活添加配置项、属性及关系；属性数据类型、填充规则均可动态定义 |
| **配置查询灵活多样** | 在线自定义多配置关联查询 |
| **细粒度权限管控** | 数据权限控制，在线配置 |
| **开放 API 服务** | 在线定义接口可读写的配置项、属性及关系，在线测试 |
| **多维度日志查询** | 数据变迁历史查询 |
| **视图管理** | IDC 规划设计、IDC 资源规划、应用架构设计、应用部署设计 |

**设计亮点：**
- 四层视图：物理层 → 逻辑层 → 应用层 → 关系
- 配置信息是 IT 运维自动化的基石
- 与自动化运维工具紧密联动

### 1.3 嘉为蓝鲸

**定位：** 基于蓝鲸 CMDB 的企业级运维平台

**核心场景：**
- 多数据中心统一运维
- 配置数据管理
- 监控管理
- 日志管理
- 统一告警管理
- 自动化巡检
- 运营服务报表

**客户案例：** 四川农信、北京移动、中国银河证券、人保科技、华夏银行、中信建投等

### 1.4 博睿 ONE

**CMDB 特点：**
- APM 驱动的 CMDB（从 Trace 数据自动发现实体和关系）
- 拓扑 = 核心可视化（全局拓扑 + 纵向下钻）
- 侧重于可观测性（指标/Trace/日志），CMDB 是支撑层

## 2. 竞品 CMDB 对比分析

| 维度 | 蓝鲸 CMDB | WeCMDB | 博睿 ONE | **我们的现状** |
|------|-----------|--------|----------|-------------|
| **模型定义** | 内置+自定义模型 | 动态模型扩展 | 固定模型(APM为主) | ✅ entity_type_def.definition JSONB |
| **属性体系** | 属性可扩展 | 属性类型+填充规则动态定义 | 固定属性 | ⚠️ attributes JSONB 自由存储，缺少元数据约束 |
| **指标体系** | 无（CMDB 本身不管指标） | 无 | 内置 APM 指标 | ⚠️ type_def.definition.metrics 有但不完整 |
| **关系模型** | 关联关系可自定义 | 关系可动态定义 | Trace 自动发现 | ✅ relationship + dimension |
| **拓扑可视化** | 业务拓扑、物理拓扑 | 四层视图 | 全局拓扑+下钻 | ✅ 刚实现三视图 |
| **数据发现** | 自动发现 + 事件推送 | 依赖外部工具 | Trace 驱动 | ✅ trace_discovery |
| **权限管理** | 用户组权限 | 细粒度数据权限 | 基础权限 | ❌ 无 |
| **事件推送** | 回调方式 | 无 | 无 | ⚠️ cmdb_event_log 有表无逻辑 |
| **审计日志** | 操作审计 | 数据变迁历史 | 无 | ❌ 无 |
| **视图管理** | 业务拓扑图 | IDC/应用架构四层视图 | 拓扑视图 | ⚠️ 拓扑刚起步 |

## 3. 我们 CMDB 的差距分析

### 3.1 已有的优势

1. **四维度实体模型**（身份/期望/观测/影响）— 业界领先的设计理念
2. **definition JSONB 收敛** — 比蓝鲸的 model 更灵活
3. **Trace 自动发现关系** — 蓝鲸/WeCMDB 都不具备
4. **健康度+风险度引擎** — 内置在 CMDB 中，竞品没有
5. **双维度关系**（横向调用链/纵向归属树）— 独特设计

### 3.2 需要补强的短板

#### 短板 1：指标体系不完善 ❌
**现状：** type_def.definition.metrics 有定义，但健康引擎只用了 3-4 个通用指标
**差距：** 没有按实体类型定义完整的指标维度和阈值

#### 短板 2：属性元数据约束缺失 ❌
**现状：** attributes 是纯 JSONB 自由存储，没有类型校验、必填检查、默认值
**差距：** 蓝鲸/WeCMDB 的属性有完整的元数据定义（类型/规则/填充方式）

#### 短板 3：关系模型不够丰富 ⚠️
**现状：** 只有 calls/depends_on/runs_on/includes 等少数关系类型
**差距：** 蓝鲸支持自定义关系类型 + 关系属性

#### 短板 4：事件推送机制不完整 ⚠️
**现状：** cmdb_event_log 表存在但无消费者
**差距：** 蓝鲸有完整的事件注册+回调推送

#### 短板 5：权限和审计完全缺失 ❌
**现状：** 无权限控制，无操作审计
**差距：** 蓝鲸有用户组权限，WeCMDB 有细粒度数据权限

## 4. CMDB 优化路线图

### Phase 4.1：对象分类的指标体系

**目标：** 每种实体类型有完整的指标维度定义

```
Service 指标体系：
├── 性能维度
│   ├── http.server.request.duration.p99 (阈值: warn=500, crit=2000)
│   ├── http.server.request.duration.p50 (阈值: warn=200, crit=800)
│   └── http.server.request.qps (阈值: 无)
├── 错误维度
│   ├── http.server.request.error_rate (阈值: warn=1%, crit=5%)
│   └── http.server.request.5xx_count (阈值: warn=10, crit=50)
├── 资源维度
│   ├── system.cpu.usage (阈值: warn=70%, crit=90%)
│   ├── system.memory.usage (阈值: warn=80%, crit=95%)
│   └── system.disk.usage (阈值: warn=80%, crit=90%)
└── 业务维度
    └── business.order.success_rate (阈值: warn=99%, crit=95%)
```

**实现：**
- 扩展 entity_type_def.definition.metrics，增加完整阈值和单位
- 新增 metric_definition 表（可选）作为指标元数据管理
- 健康引擎按类型完整计算

### Phase 4.2：维度属性元数据

**目标：** 属性有类型约束、必填检查、默认值、填充规则

```
属性定义增强：
{
  "key": "port",
  "name": "服务端口",
  "type": "int",
  "required": true,
  "default": 8080,
  "min": 1,
  "max": 65535,
  "fill_rule": "auto_discover",  // auto_discover / manual / api
  "description": "服务监听端口"
}
```

**实现：**
- 扩展 definition.attributes 的元数据 schema
- API 层增加属性校验
- 前端表单根据元数据自动生成

### Phase 4.3：关系模型增强

**目标：** 支持更多关系类型 + 关系属性 + 关系约束

```
关系增强：
- 关系属性：calls 关系增加 avg_latency/error_rate/qps
- 关系约束：Service 只能 calls Service，不能 calls Host
- 关系生命周期：自动过期 + 人工确认
- 关系来源追溯：manual/auto_discovered/imported
```

**实现：**
- 扩展 relationship 表增加更多属性字段
- relationship_type_def 增加约束规则
- 关系融合策略完善

### Phase 4.4：事件推送机制

**目标：** 实体/关系变更时主动通知下游

```
事件类型：
- entity.created / updated / deleted
- relationship.created / updated / expired
- health_level.changed (healthy → warning)
- risk_score.threshold_exceeded

推送方式：
- Webhook 回调
- Redis Streams
- 飞书通知
```

**实现：**
- 完善 cmdb_event_log 消费逻辑
- 新增事件订阅 API
- 健康引擎集成事件触发

### Phase 4.5：权限与审计（远期）

- 基础 RBAC（管理员/运维/只读）
- 操作审计日志
- 数据权限（按 biz_service 隔离）
