## 1. 全局拓扑（截图 1: 全局1.png）

### 布局特点
- **分层展示**：从上到下 = DB层 → 服务层 → 业务入口层
- **图标化节点**：每个节点是一个圆角矩形卡片，左侧有服务图标（绿色圆点 = Tomcat/服务）
- **连线样式**：灰色实线箭头，表示调用/依赖方向
- **虚线区域**：用虚线框圈出同一组/同机房的服务（如 "内网-01"、"阿里云深圳"）

### 节点信息
每个节点卡片内显示：
- 服务名（如 "orderManagement"、"service-one-test"、"managementCenter"）
- 探针状态：绿色小点 = 已安装探针
- 依赖类型标签：橙色标签显示关系类型（"数据库"、"服务"）

### 布局层次（从上到下）
```
第1层（底层DB）：centralDB, ordermanagementdb, myTestDB, Management_Center, anotherDB, myTestDB_DB2
第2层（中间服务）：中间件/RPC 调用（虚线框内）
第3层（核心服务）：orderManagement, service-one-test, managementCenter
第4层（业务入口）：CheckOut, ShippingCosts, listManagement
第5层（Web层）：Website
```

### 关键设计
- **自下而上的调用链**：DB 在底部，Web 在顶部，符合请求流向
- **分组虚线框**：同一部署环境的服务用虚线圈起来
- **节点大小统一**：所有节点等宽等高，视觉整齐

## 3. 纵向拓扑视图（截图 3: 纵向拓扑.png）

### 切换方式
- 从 "健康度" Tab 切换到 "拓扑" Tab
- 同一个抽屉面板内切换

### 布局特点
- **从上到下的树状展开**：
  - 顶层：managementCenter（核心服务）
  - 中间层：多个 Tomcat 服务（同一虚线框内）
  - 底层：两个 DB（Management_Center, myTestDB）

- **连线方向**：向下箭头，表示依赖方向
- **虚线框分组**：同一组的服务用虚线框圈起来

### 这正是主人要求的 "纵向归属树"
- 从当前服务出发，向下展开所有依赖
- DB 作为叶子节点
- 中间服务可以继续展开

## 5. 核心交互总结

| 交互 | 博睿实现 | 我们的设计 |
|------|---------|-----------|
| 全局拓扑 | 分层布局，DB→服务→Web，从下到上 | ✅ 类似，Business→Service→DB |
| 节点选中 | 蓝色高亮 + 其他变灰 + 连线高亮 | ✅ 实现此效果 |
| 抽屉详情 | 右侧 Drawer，35%宽度 | ✅ 已设计 |
| 指标 Tab | 健康度/响应时间/通量/错误/Apdex | ✅ 需要扩展 Tab |
| 纵向下钻 | 在抽屉内切换 "拓扑" Tab | ✅ 已设计 |
| 分组虚线 | 同一环境的服务用虚线框圈起来 | 🆕 新增需求 |
| 节点样式 | 圆角矩形卡片 + 左侧图标 + 内嵌标签 | 🆕 需要调整 |

## 6. 需要调整的设计点

基于博睿截图，我们的设计需要以下调整：

1. **节点样式**：从圆形改为圆角矩形卡片（更像博睿）
2. **分组虚线框**：按 biz_service 或 labels.env 分组，用虚线框圈起来
3. **布局方向**：全局拓扑从下到上（DB 在底部，Web 在顶部）
4. **抽屉 Tab 结构**：三层 Tab（一级：健康度/拓扑，二级：指标类型，三级：具体指标）
5. **选中效果**：高亮选中节点 + 相关连线，其他变灰
6. **节点信息**：卡片内显示服务名 + 探针状态 + 依赖类型标签
